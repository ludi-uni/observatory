# Observatory v1.0 仕様書

> 本書は `Observatory MCP` と `Observation Service` を **1 リポジトリ** として公開する前提の統合仕様である。  
> Discord Bot・配信 Runtime・Planner 等の上位レイヤは本リポジトリのスコープ外とする。

---

## 1. 概要

**Observatory** は LLM 向けの観測基盤である。

従来の検索 MCP が

```text
search()
crawl()
fetch()
```

を公開するのに対し、Observatory は

```text
observe()
```

という高レベルな観測インターフェースを提供する。

### 設計目標

LLM に以下を意識させない。

```text
どう検索するか
どう取得するか
どう保存するか
```

代わりに、以下だけを指定させる。

```text
何を観測したいか
```

### 設計理念

Observatory は検索サービスではない。

目的は「検索結果を返すこと」ではなく、**観測結果を生成すること**である。

将来的にエージェントが

```text
私はこう観測した
だからこう判断した
```

と説明できるよう、観測結果には **証拠（evidence）** を含める。

```text
observatory/
├── packages/
│   ├── mcp-server/          # Observatory MCP（薄いアダプタ）

├── docs/
│   └── 仕様.md              # 本書
├── examples/                # MCP クライアント設定例
└── LICENSE
```

### 2.2 スコープ

#### 含む

```text
Observatory MCP Server
Observation Service
共有型定義
Docker Compose によるローカル / OCI 起動
```

#### 含まない

```text
Discord Bot
配信 Runtime
Planner
エージェント人格制御
会話管理
```

これらは Observatory を **利用する** 上位レイヤであり、別リポジトリで管理する。

## 3. システムアーキテクチャ

```text
LLM / Agent
     │
     ▼
Observatory MCP          ← packages/mcp-server
     │  HTTP
     ▼
Observation Service      ← packages/service
     │
     ├─ Search Provider   (SearXNG)
     ├─ Fetch Provider    (Playwright)
     ├─ Extractor         (Trafilatura)
     ├─ Summarizer        (LLM or ルールベース)
     └─ Storage           (PostgreSQL)
```

### レイヤ責務

| レイヤ | 責務 |
|--------|------|
| **MCP** | ツール定義、Service への HTTP 呼び出し、レスポンス整形、MCP エラー変換 |
| **Service** | 検索・取得・抽出・要約・保存・キャッシュ・観測結果生成 |

MCP は薄いアダプタとして動作し、観測ロジックを持たない。

## 4. Observatory MCP

### 4.1 MCP Tools

#### observe

トピックを観測する。

**Input**

```json
{
  "topic": "Oracle Cloud Backup"
}
```

**Output**

```json
{
  "summary": "...",
  "confidence": 0.92,
  "sources": [
    {
      "title": "...",
      "url": "..."
    }
  ],
  "evidence": [
    {
      "source": "https://docs.oracle.com/...",
      "claim": "OCI Backup は Object Storage に保存される"
    }
  ],
  "observed_at": "2026-06-22T12:34:56Z"
}
```

**用途:** 最新情報取得、調査、質問回答、雑談ネタ取得

---

#### observe_url

URL を直接観測する。

**Input**

```json
{
  "url": "https://example.com"
}
```

**Output**

```json
{
  "title": "...",
  "summary": "...",
  "content": "...",
  "evidence": [
    {
      "source": "https://example.com",
      "claim": "..."
    }
  ],
  "observed_at": "2026-06-22T12:34:56Z"
}
```

**用途:** 記事読解、ドキュメント解析、ブログ要約

---

#### recall_observation

過去の観測履歴を取得する。

**Input**

```json
{
  "topic": "Oracle Cloud Backup"
}
```

**Output**

```json
{
  "history": [
    {
      "summary": "...",
      "confidence": 0.91,
      "observed_at": "2026-06-20T08:00:00Z"
    }
  ]
}
```

**用途:** 過去調査参照、重複観測回避、知識再利用

---

### 4.2 MCP エラーハンドリング

MCP が LLM に返すエラーは、Service の詳細エラーを集約した形式とする。

| エラーコード | 意味 |
|--------------|------|
| `SERVICE_UNAVAILABLE` | Observation Service 未接続・ヘルスチェック失敗 |
| `OBSERVATION_FAILED` | 観測処理失敗（検索・取得・抽出・要約のいずれか） |
| `TIMEOUT` | タイムアウト |

```json
{
  "success": false,
  "error": "OBSERVATION_FAILED"
}
```

Service 側の `SEARCH_FAILED` / `FETCH_FAILED` / `EXTRACTION_FAILED` は、MCP 層で `OBSERVATION_FAILED` にマッピングする。詳細は MCP のログに記録し、LLM には返さない。

---

### 4.3 MCP 設定

```env
OBSERVATION_SERVICE_URL=http://localhost:52033
OBSERVATION_API_KEY=
OBSERVATION_TIMEOUT=30
```

---

### 4.4 MCP レスポンス設計方針

LLM に返す情報は **観測結果** であり、**検索結果一覧** ではない。

良い例:

```json
{
  "summary": "...",
  "confidence": 0.89,
  "evidence": [{ "source": "...", "claim": "..." }]
}
```

悪い例:

```json
{
  "raw_search_results": [...]
}
```

## 5. Observation Service

### 5.1 責務

#### 実施すること

```text
検索
Web ページ取得
本文抽出
要約生成
信頼度算出
観測履歴保存
キャッシュ管理
観測結果返却
```

#### 実施しないこと

```text
人格制御
会話管理
エージェント制御
MCP 公開
```

---

### 5.2 コンポーネント

| コンポーネント | 初期実装 | 責務 |
|----------------|----------|------|
| Search Provider | SearXNG | 検索実行、検索結果取得 |
| Fetch Provider | Playwright | ページ取得、JS レンダリング |
| Extractor | Trafilatura | 本文抽出、広告・ノイズ除去 |
| Summarizer | OCI 小型 LLM またはルールベース | 観測結果要約、evidence 生成、信頼度算出 |
| Storage | PostgreSQL | 履歴・ページ・キャッシュ保存 |

Playwright と Trafilatura は `observation-service` コンテナ内に組み込む。

---

### 5.3 HTTP API

ベースパス: `/v1`  
認証: `Authorization: Bearer ${OBSERVATION_API_KEY}`（API キー未設定時は認証スキップ可、開発用）

#### POST /v1/observe

トピック観測。

**Request**

```json
{
  "topic": "Oracle Cloud Backup"
}
```

**処理フロー**

```text
検索 (SearXNG, Top 5)
  ↓
ページ取得 (Playwright)
  ↓
本文抽出 (Trafilatura)
  ↓
要約・evidence 生成 (Summarizer)
  ↓
保存 (PostgreSQL)
  ↓
返却
```

**Response**

```json
{
  "summary": "...",
  "confidence": 0.91,
  "sources": [
    { "title": "...", "url": "..." }
  ],
  "evidence": [
    { "source": "...", "claim": "..." }
  ],
  "observed_at": "2026-06-22T12:34:56Z"
}
```

---

#### POST /v1/observe-url

URL 観測。

**Request**

```json
{
  "url": "https://example.com"
}
```

**Response**

```json
{
  "title": "...",
  "summary": "...",
  "content": "...",
  "evidence": [
    { "source": "https://example.com", "claim": "..." }
  ],
  "observed_at": "2026-06-22T12:34:56Z"
}
```

URL キャッシュを確認し、ヒット時は再取得をスキップする（[5.6 キャッシュ](#56-キャッシュ) 参照）。

---

#### GET /v1/observation

過去観測取得。MCP の `recall_observation` に対応。

**Query**

```text
topic=Oracle Cloud Backup
```

**Response**

```json
{
  "history": [
    {
      "summary": "...",
      "confidence": 0.91,
      "observed_at": "2026-06-20T08:00:00Z"
    }
  ]
}
```

---

#### GET /health

ヘルスチェック。

**Response**

```json
{
  "status": "ok"
}
```

### 5.4 Service エラーハンドリング

| エラーコード | 意味 |
|--------------|------|
| `SEARCH_FAILED` | SearXNG 検索失敗 |
| `FETCH_FAILED` | Playwright 取得失敗 |
| `EXTRACTION_FAILED` | Trafilatura 抽出失敗 |
| `TIMEOUT` | 処理タイムアウト |

```json
{
  "success": false,
  "error": "FETCH_FAILED"
}
```

---

### 5.5 要約・信頼度

#### 要約

v1 では上位 3〜5 記事を統合し、1 つの `summary` と複数の `evidence` を生成する。

#### 信頼度（v1 簡易版）

| 要素 | 影響 |
|------|------|
| 取得成功率 | ベーススコア |
| 記事数 | 複数ソースで加点 |
| ドメイン数 | 多様性で加点 |
| 本文長 | 情報量の指標 |
| 公式サイトあり | +0.2 |
| 複数ソース一致 | +0.2 |
| 取得失敗 | -0.1（ソースごと） |

`confidence` は 0.0〜1.0 の実数。

---

### 5.6 キャッシュ

#### URL キャッシュ（`fetched_pages` テーブル）

| 項目 | 値 |
|------|-----|
| キー | `url` |
| TTL | 7 日 |

```text
URL 取得要求
  ↓
DB 確認（url + TTL）
  ↓
キャッシュあり → 利用
キャッシュなし → Playwright で再取得 → 保存
```

---

### 5.7 PostgreSQL スキーマ

#### observations

```sql
CREATE TABLE observations (
    id UUID PRIMARY KEY,
    topic TEXT NOT NULL,
    summary TEXT,
    confidence REAL,
    evidence JSONB,
    observed_at TIMESTAMP NOT NULL
);
```

#### observation_sources

```sql
CREATE TABLE observation_sources (
    id UUID PRIMARY KEY,
    observation_id UUID NOT NULL REFERENCES observations(id),
    title TEXT,
    url TEXT NOT NULL,
    content TEXT,
    fetched_at TIMESTAMP NOT NULL
);
```

#### fetched_pages

```sql
CREATE TABLE fetched_pages (
    id UUID PRIMARY KEY,
    url TEXT UNIQUE NOT NULL,
    title TEXT,
    content TEXT,
    content_hash TEXT,
    fetched_at TIMESTAMP NOT NULL,
    last_verified_at TIMESTAMP NOT NULL
);
```

#### search_queries

```sql
CREATE TABLE search_queries (
    id UUID PRIMARY KEY,
    query TEXT NOT NULL,
    query_hash TEXT NOT NULL,
    searched_at TIMESTAMP NOT NULL
);
```

## 6. MCP ↔ Service マッピング

| MCP Tool | Service Endpoint |
|----------|------------------|
| `observe` | `POST /v1/observe` |
| `observe_url` | `POST /v1/observe-url` |
| `recall_observation` | `GET /v1/observation?topic=` |

| Service Error | MCP Error |
|---------------|-----------|
| `SEARCH_FAILED` | `OBSERVATION_FAILED` |
| `FETCH_FAILED` | `OBSERVATION_FAILED` |
| `EXTRACTION_FAILED` | `OBSERVATION_FAILED` |
| `TIMEOUT` | `TIMEOUT` |
| 接続失敗 | `SERVICE_UNAVAILABLE` |

共有型は `packages/types` で定義し、MCP と Service の両方が import する。

## 7. デプロイ構成

### 7.1 Docker Compose（推奨）

```text
deploy/docker-compose.yml

services:
  observation-service   # packages/service
  searxng
  postgres
```

Playwright は `observation-service` 内にバンドルする。

### 7.2 環境変数（Service）

```env
DATABASE_URL=postgresql://...
SEARXNG_URL=http://searxng:8080
SUMMARIZER_TYPE=rule|llm
SUMMARIZER_LLM_URL=          # llm 選択時
FETCH_TIMEOUT=30
CACHE_TTL_DAYS=7
API_KEY=                     # 任意
```

### 7.3 ローカル開発

```text
docker compose up -d          # Service + 依存
pnpm --filter mcp-server dev  # MCP（stdio）
```

## 9. v1.0 完了条件

### 9.1 実装対象

```text
packages/mcp-server
  - observe / observe_url / recall_observation
  - stdio トランスポート

packages/service
  - POST /v1/observe
  - POST /v1/observe-url
  - GET /v1/observation
  - GET /health

packages/types
  - 共有型・エラーコード

deploy/
  - docker-compose.yml（observation-service + searxng + postgres）

SearXNG / Playwright / Trafilatura / PostgreSQL 連携
```

### 9.2 動作確認

```text
observe("Oracle Cloud Backup")
  ↓ 検索 (SearXNG)
  ↓ 取得 (Playwright, Top 5)
  ↓ 抽出 (Trafilatura)
  ↓ 要約 + evidence 生成
  ↓ 保存 (PostgreSQL)
  ↓ MCP 経由で LLM に返却
```

Claude Desktop または Cursor から MCP 接続し、上記フローが end-to-end で動作すること。

### 9.3 非目標（v1）

```text
observe_diff / observe_trend / observe_feed
HTTP トランスポート（MCP）
Discord / 配信 Runtime 連携
Kubernetes / OCI 本番 Terraform
```

## 11. 用語集

| 用語 | 定義 |
|------|------|
| 観測 (observation) | 検索・取得・抽出・要約を経て生成された、トピックまたは URL に対する結論 |
| 証拠 (evidence) | 観測結論を支える、ソース URL と主張のペア |
| 信頼度 (confidence) | 観測結果の確からしさを示す 0.0〜1.0 のスコア |
| ソース (sources) | 観測に使用した参照ページの一覧 |

## 付録: 旧仕様からの変更点

| 項目 | 旧仕様 | 本仕様 |
|------|--------|--------|
| リポジトリ | MCP と Service を別ドキュメント想定 | 1 モノレポに統合 |
| レスポンス | `summary` + `sources` のみ | `evidence` を追加 |
| recall API | MCP: `recall_observation` / Service: `GET /observation` | マッピング表で明示 |
| API パス | プレフィックスなし | `/v1` プレフィックスを統一 |
| DB スキーマ | `observations` に evidence なし | `evidence JSONB` 列を追加 |
